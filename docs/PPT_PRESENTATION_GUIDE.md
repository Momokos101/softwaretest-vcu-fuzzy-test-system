## 在开始讲之前——两个最基本的概念

在讲每一页之前，先把贯穿整个 PPT 的两个核心概念说清楚。

### 什么是黑盒测试

想象你拿到一台自动贩卖机，你不知道它内部的电路和程序是怎么写的。你只能从外部操作：投硬币、按按钮、看出货。你通过观察"输入什么、得到什么"来判断它有没有问题，这就是黑盒测试。

**黑盒 = 只看输入和输出，不管内部代码怎么写**

对 VCU 仿真器来说：
- 输入：供电电压 9.3V，持续 15ms
- 输出：vehicle_state=11（唤醒成功）
- 你不需要知道仿真器里面的 if-else 是怎么写的

### 什么是白盒测试

还是那台贩卖机，但这次你拿到了它的电路图和程序代码。你可以分析"哪些代码路径没有被执行到"，然后专门设计测试去覆盖那些路径。这就是白盒测试。

**白盒 = 看着代码内部结构来设计测试**

对 VCU 仿真器来说：
- 你知道仿真器内部有 state09→state10→state11 三条路径
- 你还知道有一条"卡死在 state10"的特殊路径
- 你要专门设计测试用例，确保每条路径都被执行到

## 第2页：VCU 行为仿真器

**讲什么**：被测软件具体是什么样的，有多复杂。

### 状态机是什么

VCU 不是每次收到信号都从零开始判断的，它有"记忆"。这个记忆就是状态机。

用手机举例：
- 手机有锁屏、使用中、关机三种状态
- 按电源键才能从关机到锁屏，不能直接从关机跳到使用中
- 状态机就是这样的规则：当前在哪个状态 + 收到什么操作 = 下一个状态

我们的 VCU 仿真器有三个状态：
- **state09**：VCU 休眠，相当于手机关机
- **state10**：VCU 初始化中，相当于手机开机过程
- **state11**：VCU 正常运行，相当于手机使用中

**已知缺陷**：北汽原始文档记录了一个真实 bug——频繁快速地开关机，VCU 会卡在 state10 出不来（就像手机反复重启最后卡在开机界面）。我们在仿真器里模拟了这个缺陷。

### 7路唤醒 + 3路休眠是什么

VCU 可以被7种不同的信号唤醒（w1~w7），比如：
- w1：供电电压超过 9V 持续 10ms 以上（像充电器插上了）
- w2：CAN 总线收到特定报文（像收到网络指令）
- w6：车门口盖打开（检测到物理动作）

但进入休眠必须同时满足3个条件（h1 AND h2 AND h3），缺一不可。这个设计本身就创造了很丰富的测试场景。

### 5个输出字段是什么，分别代表什么含义

仿真器每次返回的5个字段，每个都对应 VCU 真实硬件上的一个可观测量：

**① vehicle_state（整车 State 状态）**
VCU 当前处于哪个状态，是最核心的输出字段。
- 值=9：VCU 在休眠
- 值=10：VCU 在初始化（正常情况是过渡态，卡死时会一直停在这里）
- 值=11：VCU 正常运行

**② vehicle_mode（整车模式）**
整车当前的工作模式，在我们的仿真器里固定返回 2，不影响测试判断，主要是保持和真实系统输出格式一致。

**③ power_current（功耗电流）**
VCU 当前消耗多少电流。
- 正常运行：≤0.05A（低功耗）
- 卡死状态：>0.2A——北汽原始文档里特别提到"控制器不休眠电流始终大于 200mA"，这正是卡死 bug 的症状之一。我们在仿真器里复现了这个现象。

**④ bus_message_flag（总线报文发送标志位）**
CAN 总线有没有在正常发送报文。
- 值=1：VCU 在正常工作，总线通信正常
- 值=0：VCU 未唤醒或处于异常状态，总线停发

**⑤ pdcu_wake_reason（PDCU 唤醒原因）**
是哪一路信号把 VCU 唤醒的，取值 1-7 分别对应 w1-w7，0 表示没有唤醒或唤醒失败。这个字段让我们能精确验证"发了 w1 信号，唤醒原因应该是 1"，而不是只知道唤醒成功了。

---

### 5个字段和四种结果类型的对应关系

| 结果类型 | vehicle_state | power_current | bus_flag | wake_reason | 触发条件 |
|---------|--------------|---------------|---------|-------------|---------|
| **expected** | 11 | ≤0.05A | 1 | 1~7 | 一切正常 |
| **error** | 9或10（未跳转） | ≤0.05A | 0 | 0 | 信号无效/条件不足 |
| **stuck** | 10（卡住） | >0.2A | 0 | 0 | 快速连续唤醒触发 |
| **new_state** | 任意（不匹配上面三种） | 任意 | 任意 | 任意 | 发现未预料的输出 |

**判断逻辑**：工具拿到仿真器的返回值后，先和 expected_results 比——如果全部字段匹配，就是 expected；再和 error\[0\] 比——如果匹配，就是 error；再和 error\[1\] 比——如果匹配，就是 stuck；都不匹配，就是 new_state，说明发现了新的边界情况，会触发下一轮变异测试。

---

### 卡死缺陷的判断逻辑是怎么实现的

仿真器内部维护一个"最近唤醒历史列表"，每次收到唤醒请求时，按以下步骤判断：

```
第一步：把5秒以前的历史记录清掉（只看最近5秒内的）
第二步：数一数最近5秒内发生了几次唤醒
第三步：如果 ≥3次，计算相邻两次之间的间隔
第四步：如果所有间隔都 <1秒，判定为"快速连续上下电"
第五步：返回 stuck 结果
    - vehicle_state = 10（卡在初始化）
    - power_current > 0.2（复现北汽原始 bug 的功耗表现）
    - actual_duration = est_time × 3（比预期时间超出3倍）
```

**为什么这样设计**：北汽原始文档描述的 bug 触发场景是"频繁上下电"，没有给出精确的触发条件（毕竟是真实硬件上复杂的底层死循环问题）。我们把"3次以上、间隔小于1秒"作为仿真器里的近似触发条件，这是一个合理的简化，在答辩时可以直接说"参考了原始 bug 报告的描述，以快速上下电序列作为触发模拟"。

## 第3页：AutoTestDesign Tool

**讲什么**：我们的工具实现了 Assignment 要求的哪些功能。

**怎么讲**：

> Assignment 要求实现一系列功能（FR 1.0 到 FR 7.0），其中 FR 1.0/1.1/2.0/3.0/6.0 是必须实现的，FR 4.0/5.0/7.0 是加分项。
>
> 我们全部必须项都实现了，三个加分项也都纳入了方案：FR 4.0 是白盒状态机建模，FR 5.0 是用 LLM 自动生成预期结果，FR 7.0 是两轮模糊测试反馈。
>
> 底部这个流程条是 Assignment 特别强调的"Mainly"流程：工具必须支持测试人员在每个阶段手动介入修改，不能是纯自动的黑箱。我们在每个步骤都支持人工编辑。

**FR 2.0 风险分析怎么解释**：

就是给每条需求打一个优先级分数。打分依据是：这个需求如果有 bug 会造成多大危害？有多大概率出现 bug？两者相乘得到风险等级。REQ-012（卡死缺陷）因为有北汽真实 bug 报告，直接评为 High。

## 第4页：测试计划与测试设计流程

**讲什么**：我们打算怎么测，测多少，怎么执行。

### 六个测试套

我们把 82 条测试用例分成 6 组：

| 套件 | 用一句话解释 |
|------|------------|
| Suite A | 对7路唤醒信号各自测"有效/无效/时序不够"三种情况 |
| Suite B | 专门测信号的边界值（比如9.0V到底算不算有效）|
| Suite C | 穷举休眠的8种条件组合 |
| Suite D | 测状态机的每一条跳转路径 |
| Suite E | 专门触发卡死缺陷 |
| Suite F | 测响应时间是否超标 |

### 两轮模糊测试是什么

**用一个生活类比**：

第一次摸黑走进一个陌生房间，你会随机走动，撞到家具了（发现障碍）就记下来。第二次再进去，你重点围绕上次撞到的地方多走几步，把那片区域摸清楚。

模糊测试就是这个逻辑：
- 第一轮：按初始用例随机变异测试，记录发现的新状态
- 第二轮：围绕第一轮发现的新状态，生成更多相似的测试用例，找到更多边界

**执行流程**：

> 每条用例执行时，工具先发 reset 让仿真器回到初始状态，然后发 simulate 请求，仿真器同步返回结果。工具拿到结果后跟预期比较，决定下一步操作。

## 第5页：详细测试设计——黑盒 + 白盒技术

这是整个 PPT 里技术含量最高的一页，我们来逐一解释每种技术。

---

### 黑盒技术①：等价类划分（Equivalence Partitioning，EP）

**核心思想**：把所有可能的输入值分成几个"类"，同一个类里的值对软件来说效果是一样的，只需要测一个代表值就够了。

**生活类比**：
你要测一个"年龄验证"功能（成年人才能进）。
- 有效类：18-120岁（都会通过）
- 无效类1：0-17岁（都会拒绝）
- 无效类2：负数或非数字（都会报错）

你不需要测 18、19、20、21……每个值，每个类里测一个就够了。

**我们的 EP 怎么做**：

对供电电压 w1（触发条件：>9V 持续≥10ms），划分了三个等价类：
- 有效类：电压>9V 且 持续≥10ms → 应该唤醒成功
- 无效类1（值不足）：电压≤9V → 应该唤醒失败
- 无效类2（时序不足）：电压>9V 但 持续<10ms → 也应该唤醒失败

注意我们有**两个维度的无效类**，这是因为 w1 的触发条件本身是两个条件的组合（值 AND 时序），所以比普通的 EP 更丰富。

---

### 黑盒技术②：边界值分析（Boundary Value Analysis，BVA）

**核心思想**：bug 最容易出现在边界附近。比如"≥9V"这个条件，8.9V、9.0V、9.1V 是最容易写错的地方（比如写成了 >9V 而不是 ≥9V），所以要专门测边界点和它的上下各一个值。

**标准做法**：对每个边界，测三个值：
- 边界值 - 1（刚好在边界下方）
- 边界值本身
- 边界值 + 1（刚好在边界上方）

**我们的 BVA**：

我们有两个轴：

电压轴（边界在 9V）：
- 8.9V → error（不够）
- 9.0V → error（刚好不够，≥9V 才行）
- 9.1V → expected（够了）

时序轴（边界在 10ms）：
- 9ms → error（持续不够）
- 10ms → expected（刚好够）
- 11ms → expected（够了）

这样每个信号就有了两组边界用例，比只测一个边界要严格得多。

---

### 黑盒技术③：决策表（Decision Table）

**核心思想**：当一个功能需要多个条件同时满足时，把所有可能的条件组合列成表格，确保每种组合都测到了。

**生活类比**：
一个贷款申请，需要"收入≥5000" AND "信用分≥600" AND "有工作证明"才能批。
三个条件各有是/否两种情况，组合一共 2×2×2=8 种。决策表就是把这8种全列出来，搞清楚每种情况下结果是什么。

**我们的决策表**：

VCU 进入休眠需要 h1 AND h2 AND h3 同时=1，三个二值条件，一共8种组合：

| h1 | h2 | h3 | 预期 |
|----|----|----|------|
| 0  | 任意 | 任意 | 不休眠（前4种组合） |
| 1  | 1  | 1  | 休眠成功 |

只有最后一种情况才能成功休眠。这是非常典型的决策表场景，因为条件是 AND 关系。

---

### 白盒技术④：状态转移测试（State Transition Testing）

**核心思想**：把软件内部的状态机画出来，确保每一条状态转移路径都被测试到。

**区别于黑盒的地方**：黑盒测试只看输入输出，状态转移测试是因为我们知道内部有 state09/10/11 这个结构，所以专门针对这个结构设计测试。

**我们设计了5条路径**：

```
T-001 正常唤醒：  state09 → state11（发有效信号）
T-002 正常休眠：  state11 → state09（发h1+h2+h3）
T-003 休眠失败：  state11 → state11（只发h1+h2，缺h3）
T-004 唤醒失败：  state09 → state09（发无效信号）
T-005 卡死触发：  state09 → state10（快速连续3次唤醒）
```

"All-Transitions 覆盖准则"就是要求这5条路径全部都要有测试用例覆盖到，一条都不能漏。

---

### 白盒覆盖目标汇总

- **All-States**：state09/10/11 三个状态都必须被执行到（Suite D）
- **All-Transitions**：5条路径全部覆盖（Suite D + E）
- **Branch Coverage**：7路唤醒信号的有效/无效两个分支都要测（Suite A）
- **Stuck Path**：T-005 卡死路径专项覆盖（Suite E）

## 第6页：我们的疑问

### Q1：被测对象选择是否合适

**背景**：Assignment 示例给的目标应用是 web app、计算器、登录模块等，我们选的是 VCU 仿真器（汽车嵌入式系统模拟）。

**我们的理由**：仿真器基于北汽真实数据库重建，有真实 bug 记录，不是随手捏造的。但需要确认老师认可这个选择。

### Q2：人造 stuck bug 对白盒测试的价值

**背景**：state10 卡死这个 bug 来自北汽真实文档，但在我们的仿真器里是我们主动写进去的（特定序列触发 stuck 返回）。

**疑问**：这种"明知道有个 bug，专门设计测试去发现它"的方式，在白盒测试评分里算不算有效？

### Q3：两轮模糊测试反馈对应 FR 7.0 吗

**背景**：FR 7.0 要求"基于风险或覆盖效率优化测试套"。我们的实现是：第一轮测完，把发现的新状态存起来，第二轮围绕新状态再生成变异用例。

**疑问**：这算不算"基于覆盖效率优化"？老师怎么理解 FR 7.0 的范围？

### Q4：GAN 如何归类

**背景**：我们之前的系统有一个 GAN 模块，可以生成 CC2 电压的时序序列数据。

**疑问**：GAN 是测试数据生成工具，不是 ISO 29119-4 的黑盒/白盒测试技术。它应该归入 FR 3.0（测试用例生成）还是 FR 7.0（测试套优化）？如果两者都不对应，它对评分有没有实质贡献？
